# Audio Support in Docker Extension VNC

This document explains the audio system used in the Docker Extension VNC, including how it works under the hood and how you can configure your own custom Dockerfiles to fully support both Audio Output and Audio Input from scratch.

## Overview

The audio system treats **Audio Output** and **Audio Input** as two entirely separate entities, meaning you can enable or disable them independently.

- **Audio Output**: Sound generated by the applications inside the container, streamed *out* to the user's browser (e.g., listening to a YouTube video or system alerts).
- **Audio Input**: Sound captured from the user's browser microphone, streamed *into* the container (e.g., using a web mic, voice chat, or recording software).

The audio transport between the container and the browser is facilitated natively by **PulseAudio**, **FFmpeg** (using the low-latency SRT protocol), and the **vnc-viewer-proxy**.

> 💡 **Quick Start:** We have provided a fully functioning, standalone `Dockerfile` and `start.sh` script demonstrating this architecture. Check out the [example](example) directory to see the complete code!

---

## How it Works

### Audio Output (Container ➡️ Browser)
1. **PulseAudio** creates a virtual sink named `virtual_sink`.
2. Applications in the container play sound to this sink by default.
3. An **FFmpeg** loop continuously reads from `virtual_sink.monitor` and serves it as an SRT listener on port `7900`.
4. The `vnc-viewer-proxy` connects to port `7900`, retrieves the stream, and serves it to the browser via WebRTC.

### Audio Input (Browser ➡️ Container)
1. The user's browser captures microphone audio and sends it to the `vnc-viewer-proxy`.
2. The proxy exposes this incoming stream via SRT on port `8890`.
3. An **FFmpeg** loop in the container continuously reads from `srt://vnc-viewer-proxy:8890?streamid=read:mic`.
4. It pipes this incoming audio to a PulseAudio sink (`mic_sink`), which is remapped to a virtual source (`virtmic`).
5. Applications in the container use `virtmic` as their default microphone.

## Step-by-Step Guide for Custom Dockerfiles

To explicitly declare audio support in a completely custom Dockerfile (without relying on our base image entrypoints), you must install dependencies, define environment variables for the proxy, configure PulseAudio, and manually launch the audio streams in your startup script.

### Step 1: Install Dependencies

You need to ensure that **PulseAudio** and **FFmpeg** are installed in your container so audio is processable into a protocol the VNC proxy understands.

```dockerfile
# Example for Debian/Ubuntu based images:
USER root

# Install dependencies required for audio setup
RUN apt-get update && \
    DEBIAN_FRONTEND=noninteractive apt-get install -y --no-install-recommends \
    pulseaudio \
    pulseaudio-utils \
    ffmpeg && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*
```

*Note: The package name `pulseaudio-utils` is used on Debian/Ubuntu/Fedora to supply the `pactl` command. If you're on Alpine, you can install `pulseaudio` using `apk add pulseaudio`. If you're on Arch Linux, `pactl` ships as part of `libpulse`.*

### Step 2: Set Environment Variables

The VNC extension proxy determines if a container supports audio features based on the presence of two environment variables: `VNC_EXTENSION_AUDIO_OUTPUT` and `VNC_EXTENSION_AUDIO_INPUT`.

Set these variables in your `Dockerfile` for the extension to wire things up correctly at boot time:

```dockerfile
# Must be set to true for the VNC Extension Proxy to know you support these features.
ENV VNC_EXTENSION_AUDIO_OUTPUT=true
ENV VNC_EXTENSION_AUDIO_INPUT=true
```

*(Alternatively, the proxy also natively looks for `/etc/.vnc_ext_audio_output` and `/etc/.vnc_ext_audio_input` marker files, but defining environment variables is often the most direct path).*

### Step 3: Write an Entrypoint / Startup Script

A Docker container expects a single running process (usually your `CMD` or `ENTRYPOINT`). Because the audio system relies on persistent background processing, you will need to add a startup script (e.g. `start.sh`) that spins up the PulseAudio daemon, its virtual devices, and the FFmpeg loops in the background before finally spawning your Desktop Environment (or VNC Server).

Here is a full breakdown of what your custom startup script needs to include.

#### A. Configure and Start PulseAudio

First, make sure PulseAudio is running and reachable. *It is highly recommended to run PulseAudio as a non-root user (e.g. `vncuser`).*

```bash
#!/bin/bash
set -eo pipefail

echo "🔊 Starting PulseAudio..."

# Standard XDG environment setup for PulseAudio sockets
export RUNTIME_DIR="/tmp/runtime"
export PULSE_RUNTIME_PATH="$RUNTIME_DIR/pulse"
export PULSE_SERVER="unix:$PULSE_RUNTIME_PATH/native"

mkdir -p "$PULSE_RUNTIME_PATH"
chmod 700 "$PULSE_RUNTIME_PATH"

# Configure daemon and client to run smoothly in a container without systemd
mkdir -p "$HOME/.config/pulse"
cat > "$HOME/.config/pulse/daemon.conf" <<EOF
daemonize = yes
system-instance = no
EOF

cat > "$HOME/.config/pulse/client.conf" <<EOF
autospawn = no
default-server = $PULSE_SERVER
EOF

# Start PulseAudio daemonized
pulseaudio --daemonize=yes --log-target=stderr --realtime=no

# Wait slightly for the socket to appear
sleep 2 
```

#### B. Create the Virtual Audio Devices

Next, you must carve out the virtual sinks that the FFmpeg pipelines will wire themselves into.

```bash
echo "🎛️ Creating PulseAudio Virtual Devices..."

# Create the Audio Output Sink (Apps will play audio here)
pactl load-module module-null-sink \
    sink_name=virtual_sink \
    sink_properties=device.description=Virtual_Sink

# Set this as the default so apps output here automatically
pactl set-default-sink virtual_sink

# Create the Audio Input (Microphone) Sink and Link It
# (The incoming FFmpeg stream plays into mic_sink, which we remap as a microphone source)
pactl load-module module-null-sink \
    sink_name=mic_sink \
    sink_properties=device.description=Virtual_Mic_Input

pactl load-module module-remap-source \
    master=mic_sink.monitor \
    source_name=virtmic \
    source_properties=device.description=Virtual_Microphone

# Set this as the default so apps listen here automatically
pactl set-default-source virtmic
```

#### C. Spawn the Output and Input FFmpeg Loops

Since Docker containers can crash or proxy networks might hiccup, wrap your FFmpeg stream logic inside unbreakable `while true` loops. Place these in the background using `&`.

```bash
echo "🚀 Starting FFmpeg background loops..."

# --- 1. Audio Output Supervisor (Container > Browser) ---
# Listens on SRT port 7900 for the proxy to fetch audio
start_audio_output_loop() {
  while true; do
    ffmpeg -loglevel error -nostdin \
      -f pulse -i "virtual_sink.monitor" \
      -c:a libopus -b:a 96k -vbr off -compression_level 5 \
      -frame_duration 10 -application lowdelay \
      -f mpegts -pes_payload_size 0 -flush_packets 1 \
      "srt://0.0.0.0:7900?mode=listener&latency=0"
    sleep 1
  done
}
start_audio_output_loop &

# --- 2. Mic Input Supervisor (Browser > Container) ---
# Connects to the proxy (vnc-viewer-proxy) on port 8890 and pipes it to the mic sink
start_mic_loop() {
  export SRT_LOG_LEVEL=error
  while true; do
    # Only actually attempt to pull data if the proxy is reachable
    if ! ffmpeg -v quiet -t 1 -i "srt://vnc-viewer-proxy:8890?streamid=read:mic" -f null - 2>/dev/null; then
      sleep 2
      continue
    fi

    ffmpeg -loglevel error -nostdin -v quiet \
      -i "srt://vnc-viewer-proxy:8890?streamid=read:mic&latency=0" \
      -c:a pcm_s16le -ar 48000 -ac 2 \
      -f pulse "mic_sink"

    sleep 2
  done
}
start_mic_loop &
```

> **Alternative Frameworks:** We use low-latency UDP streams over SRT to ensure Voice/Audio stays perfectly in sync with the VNC stream. Using tools other than FFmpeg to handle generic SRT isn't heavily supported, so this setup is highly recommended!

#### D. Start your Application

After the background loops are safely executing, finish your startup script with `exec` to spin up your target desktop or server, retaining the foreground process.

```bash
# Start your VNC server or desktop environment
echo "🖥️ Starting VNC Server..."
exec tigervncserver :0 -fg -geometry 1920x1080
```

### Step 4: Finalizing the Dockerfile

Putting it all together, point your custom `Dockerfile` to your newly created startup script.

```dockerfile
# (Assuming setup from Step 1 has completed)
COPY start.sh /home/vncuser/start.sh
RUN chmod +x /home/vncuser/start.sh

# Change to user mode
USER vncuser
WORKDIR /home/vncuser

CMD ["/home/vncuser/start.sh"]
```

Build the container, and Audio Input and Audio Output will flawlessly integrate with your extension!
